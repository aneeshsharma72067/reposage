# Agentic Engineering Workflow AI — Development Status

**Project:** Agentic Engineering Workflow AI
**Last Updated:** March 2026
**Architecture Type:** Event-driven distributed analysis system

---

# 1. Project Overview

Agentic Engineering Workflow AI is a developer productivity platform that analyzes repository changes and generates engineering insights based on commit activity.

The system continuously monitors connected GitHub repositories, processes code change events asynchronously, runs analysis rules on commit diffs, and surfaces findings in a dashboard.

The architecture is designed to support both deterministic rule-based analysis and future AI-powered code analysis.

---

# 2. High Level Architecture

The system currently follows an **event-driven distributed architecture**.

```text
Developer Push
      │
      ▼
GitHub Webhook
      │
      ▼
API Server (Fastify)
      │
Store Event + Create AnalysisRun
      │
Publish Job
      ▼
Redis Queue
      │
      ▼
Worker (Analyzer Service)
      │
Fetch Commit Diff from GitHub
      │
Run Analysis Rule Engine
      │
Generate Findings
      ▼
PostgreSQL Database
      │
      ▼
Frontend Dashboard (Next.js)
```

This architecture ensures that heavy processing never blocks the API server and allows the system to scale horizontally.

# 4. Backend Implementation

## 4.1 API Server

The API server is implemented using **Fastify** and acts as the system control plane.

Responsibilities include:

- GitHub OAuth authentication
- GitHub App installation flow
- Repository synchronization
- Webhook ingestion
- Event persistence
- Analysis job publishing
- Serving dashboard APIs

The API does **not perform analysis itself**.
It only stores events and publishes jobs to Redis.

---

## 4.2 GitHub Integration

The system integrates with GitHub through a **GitHub App**.

Implemented features include:

- GitHub OAuth login
- GitHub App installation
- Repository synchronization
- Installation token generation
- Secure API access using installation tokens

The GitHub App private key is used to generate JWTs, which are exchanged for installation access tokens.

These tokens allow the worker to fetch repository data securely.

---

## 4.3 Webhook System

GitHub sends events such as:

- Push events
- Pull request events (future)

The webhook module performs:

1. Signature validation
2. Event parsing
3. Event storage
4. AnalysisRun creation
5. Job publishing to Redis

This ensures webhook handling is fast and reliable.

---

# 5. Queue System

The system uses **Redis + BullMQ** for asynchronous job processing.

## Queue Flow

```text
Webhook received
      │
Create Event
      │
Create AnalysisRun (PENDING)
      │
Publish job to Redis
```

Jobs are stored in Redis until processed by the worker.

---

## Queue Configuration

The queue is configured with production-ready settings:

- Retry attempts: **3**
- Retry backoff: **exponential**
- Worker concurrency: **5 jobs in parallel**

This ensures resilience against temporary failures.

---

# 6. Worker Service

The worker service runs inside:

```text
workers/analyzer
```

The worker listens to the Redis queue and processes analysis jobs.

Responsibilities:

- Consume jobs from Redis
- Fetch commit diff from GitHub
- Build analysis context
- Run rule engine
- Generate findings
- Update repository health
- Update AnalysisRun status

# 7. Database Schema

The system uses **PostgreSQL with Prisma ORM**.

Key models include:

### User

Stores authenticated GitHub users.

### GithubInstallation

Represents a GitHub App installation.

### Repository

Stores synced repositories.

Status values include:

- IDLE
- ANALYZING
- HEALTHY
- ISSUES_FOUND

---

### Event

Represents a webhook event.

Stores:

- event type
- payload
- processing status

---

### AnalysisRun

Represents a single analysis execution.

States include:

- PENDING
- RUNNING
- COMPLETED
- FAILED

---

### Finding

Represents an issue or insight generated by analysis.

Each finding contains:

- type
- severity
- title
- description
- metadata

Severity levels include:

- INFO
- WARNING
- CRITICAL

# 8. Analysis Engine

The analysis system has been refactored into a **rule-based engine**.

Instead of hardcoding logic in a single service, the system executes independent rule modules.

---

## Rule Engine Flow

```text
Commit Diff
    │
    ▼
Analysis Context
    │
    ▼
Rule Engine
    │
    ├── Large Commit Rule
    ├── Sensitive File Rule
    └── Large Diff Rule
    │
    ▼
Generated Findings
```

Rules run in **parallel** to maximize performance.

---

## Example Rules

### Large Commit Rule

Triggers when a commit changes more than a threshold number of files.

### Sensitive File Rule

Triggers when files such as:

.env
config
schema
routes

are modified.

### Large Diff Rule

Triggers when a commit modifies more than a threshold number of lines.

---

# 9. Commit Analysis

When the worker processes a job:

1. It retrieves the webhook event.
2. Extracts the commit SHA.
3. Generates a GitHub installation access token.
4. Fetches commit details from the GitHub API.
5. Builds an analysis context.
6. Runs the rule engine.

This enables analysis of **real code changes**.
